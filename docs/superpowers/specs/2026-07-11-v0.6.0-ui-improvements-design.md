# SoundCloud Desktop 0.6.0 — UI Improvements Design

Date: 2026-07-11
Status: Approved (pending spec review)

## Context

0.6.0 already lands one bug fix (silent stall + Discord ghost track on failed
track transitions — see `src/player/playbackRecovery.ts`, `audioEngine.ts`,
`store.ts`). This spec covers the UI improvements bundled into the same release.

The like/repost write backend already exists (`window.sc.likeTrack`,
`likedTrackIds`, the global `useLikes` store in `src/lib/likes.ts`, and the
debounced `useToggleSync` hook). `TrackView` already demonstrates the full
like-toggle pattern. Most of this work is reusing those pieces in new places.

## Scope (A–G)

### A. Like button on current track — PlayerBar + TrackRow

**Reusable hook** `src/lib/useTrackLike.ts`, single source of truth:

- Reads liked state from the global `useLikes` store (`ids.has(track.id)`), so
  every heart in the app stays in sync.
- Wraps `useToggleSync` for optimistic, debounced, coalesced writes.
- On toggle: calls `window.sc.likeTrack(id, next)`, updates `useLikes`, and
  dispatches `sc:likes-changed` (the existing event other views listen to).
- On failure: `useToggleSync` reverts; show a toast.
- Accepts `track: Track | null`; a null track disables the toggle.

Signature: `useTrackLike(track: Track | null) => { liked: boolean; toggle: () => void }`.

**PlayerBar** — heart button in the left "now playing" cell, to the right of the
title/artist block. Filled + accent when liked. Must be a sibling of the
existing now-playing `<button>` (no nested buttons); wrap both in a flex row.

**TrackRow** — heart toggle button in the hover-actions cluster, immediately
**left of the `+`** (`Plus`) button. Visibility rule: shown on row hover; when
the track is liked it stays visible + accent even without hover (Spotify-style,
so you can scan what you've liked). The existing per-row heart that renders
`likesCount` is a *stat* (how many people liked it) and stays unchanged — the
new control is the *action*.

### B. Like in NowPlaying (full-screen)

Add the like action to the full-screen NowPlaying view (`src/ui/NowPlaying.tsx`)
using the same `useTrackLike` hook, placed near the track title/controls, so the
"focus" view is consistent with the bar. Repost/add-to-playlist are out of scope
for 0.6.0 (future).

### C. Progress-bar jitter fix

`PlayerBar` non-waveform fallback uses `transition-[width] duration-1000
ease-linear` while `timeupdate` fires ~4×/s, so the 1s linear tween lags and
overshoots, and stutters on seek. Fix: reduce the transition to a short duration
(or drive width directly from position with a ~250ms ease), and suppress the
transition entirely while dragging/seeking (a `dragging`/seek flag already
exists). Waveform path is unaffected.

### D. Keyboard shortcuts: `L` (like) + `M` (mute)

Extend `src/ui/useKeyboardShortcuts.ts`:

- `KeyL` → toggle like on the current track (no-op if nothing playing). Reuse the
  same like write path (call through the store/`useLikes` + `likeTrack` +
  dispatch `sc:likes-changed`) so all hearts update.
- `KeyM` → toggle mute. Mirror `VolumeControl`'s mute logic (remember last
  non-zero volume, restore on unmute). Consider centralizing mute so the button
  and shortcut share one implementation.

Preserve the existing guard that ignores keys while typing in
input/textarea/contenteditable.

### E. Context menu on PlayerBar artwork + "go to current track"

- Right-click (contextmenu) on the PlayerBar now-playing artwork/title opens the
  same context menu `TrackRow` builds (play next, add to queue, add to playlist,
  open track, go to artist, open on SoundCloud, copy link) for the current
  track, via the existing `useContextMenu` store. Extract the menu-item builder
  so it isn't duplicated between `TrackRow` and here.
- "Go to current track": a control that navigates to / scrolls the current
  track into view. Minimum viable: navigate to the current track's page
  (`/track/:id`) — reuse existing routing. (Scroll-into-view within an arbitrary
  list is out of scope; navigation is enough for 0.6.0.)

### F. Persist shuffle + repeat across sessions

The player store resets `shuffle`/`repeat` on launch. Persist them (localStorage
or the existing session persistence in `src/player/session.ts`) and restore on
init. Keep it minimal: store the two values, read them when the store
initializes.

### G. Clickable playlist creator → artist profile

Data gap: `Playlist` (`electron/sc/types.ts`) has only `user: string`, no id.

- Add `userId?: number` to the `Playlist` interface.
- Populate it in `normalizePlaylist` via `num(user.id)`.
- `CollectionHeader` currently takes `meta` as a plain string. Add an optional
  way to render the creator name as a link: e.g. a `creatorId?: number` +
  `creatorName?: string` prop pair (or a `meta` ReactNode). When `creatorId` is
  present, render the creator name as a link to `/artist/:creatorId`; otherwise
  fall back to the plain string. `PlaylistView` passes the new fields.
- Guard: if `userId` is missing (older/edge data), render plain text, not a dead
  link.

## Out of scope (future releases)

Crossfade/gapless, `Ctrl+K` search focus, playlist track drag-reorder, shuffle-
play button, playback speed, sidebar playlist context menu, mini-mode, sleep
timer, recents view, live search, help overlay, NowPlaying queue reorder.

## Testing

- `useTrackLike`: unit-test the pure decision/side-effect wiring where possible
  (mock `window.sc.likeTrack` and the `useLikes` store); assert optimistic flip,
  revert on failure, and event dispatch.
- Shortcuts `L`/`M`: unit-test the key→action mapping (the handler is pure given
  a mocked store), including the typing-field guard.
- `normalizePlaylist`: extend existing normalize tests to assert `userId` is
  populated and absent-id yields `undefined`.
- Mute centralization: unit-test remember-last-volume/restore logic.
- DOM-heavy wiring (PlayerBar/NowPlaying/TrackRow rendering) is verified by
  typecheck + lint + manual smoke, consistent with the existing test strategy
  (node env, pure logic only).

## Risks / notes

- 161-row playlists render one `useToggleSync` per row — lightweight state/refs,
  acceptable; matches the existing per-row heart-count rendering.
- Adding the hover heart must not collide with the play-count/like-count columns
  that hide below 560–620px; verify layout at narrow widths.
- Not a git repository, so the "commit the design doc" step is skipped.
