# SoundCloud Desktop 0.7.0 — Playlist Reorder, Shuffle Play & Live Search

Date: 2026-07-11
Status: Approved (pending spec review)

## Context

0.7.0 adds three user-facing features to the existing SoundCloud desktop client.
The like/playlist write backend already exists (`electron/sc/api.ts` with a
private `putPlaylist(id, patch)` used by `addToPlaylist`/`removeFromPlaylist`/
`renamePlaylist`); the player store (`src/player/store.ts`) already has
`shuffle`/`shuffled()`; and the queue drag-reorder pattern already exists in
`src/ui/QueuePopover.tsx` (HTML5 drag events). This work reuses those pieces.

## Scope (A–C)

### A. Drag-reorder playlist tracks (persists to SoundCloud)

Owner-only. On the playlist page, an owner can drag a track row to a new
position. Rows reorder visually during the drag; **on drop the new order is
persisted to SoundCloud** and kept optimistically (revert + toast on failure).

- **Backend:** add `reorderPlaylist(playlistId, orderedTrackIds)` to `SoundCloudApi`
  — a thin wrapper over the existing private `putPlaylist(playlistId, { tracks:
  orderedTrackIds })`. Wire it through IPC (`SC_REORDER_PLAYLIST`), the main
  handler (`scIpc.ts`), and `preload.ts` as
  `window.sc.reorderPlaylist(id, ids)`.
- **UI:** `TrackList` gains an optional `onReorder?(from: number, to: number)`
  prop. When present, each `TrackRow` becomes draggable (HTML5 DnD, mirroring
  `QueuePopover`'s `onDragStart`/`onDragEnter`/`onDragOver`/`onDragEnd` with a
  drop indicator). When absent (the common case — non-owner lists, search,
  artist, etc.), rows are not draggable and behavior is unchanged.
- **State:** `PlaylistView` holds the displayed tracks in local state seeded from
  the fetched playlist. On drop it computes the new order, updates local state
  optimistically, and calls `window.sc.reorderPlaylist(pid, newIds)`. On failure
  it reverts to the previous order and shows an error toast. Only wired when
  `isOwner`.
- **Guard:** dragging is enabled only for owner playlists (`onReorder` passed
  only when `isOwner`).

### B. Shuffle play button (all collection headers)

`CollectionHeader` gains an optional `onShufflePlay?()` prop. When present it
renders a shuffle button next to the existing "Tocar" button. The views that
already pass `onPlay` — `LikesView`, `MixView`, `PlaylistView` — also pass
`onShufflePlay`. (Charts/Artist views use their own layouts without
`CollectionHeader` and are out of scope for this button.)

- **Player:** add `playShuffled(tracks: Track[])` to the store — sets
  `shuffle: true`, builds a shuffled queue via the existing `shuffled()` helper,
  and plays from index 0. Persists the shuffle pref via the existing
  `savePlaybackPrefs` path (consistent with `toggleShuffle`).
- **UI:** the shuffle button calls `playShuffled(tracks)`. Styling mirrors the
  existing header play button (secondary/outline treatment so "Tocar" stays the
  primary action).

### C. Live search + Ctrl+K

- **Live search:** `TopBar`'s search input debounces (~300ms) and navigates to
  `/search?q=<query>` as the user types, using `navigate(..., { replace: true })`
  so keystrokes don't flood browser history. Submitting (Enter) still navigates
  immediately. The input value initializes from and stays in sync with the URL
  `q` param, so opening `/search?q=x` shows `x` in the box.
- **Ctrl+K:** a global shortcut focuses the search input. Implemented via a
  window `CustomEvent('sc:focus-search')` dispatched from the keyboard handler;
  `TopBar` listens and focuses (and selects) its input ref. `Ctrl+K` is added to
  the pure `shortcutAction` mapper (returns `{ type: 'focus-search' }`), and
  `useKeyboardShortcuts` dispatches the event. The typing-field guard already
  prevents interfering while typing elsewhere; `Ctrl+K` should still work from
  within the search box itself (it's the target), so the handler must allow this
  specific combo even when focus is in an input — see Ambiguity Resolution.

## Ambiguity Resolution

- **Ctrl+K while typing:** `useKeyboardShortcuts` currently returns early when the
  event target is an INPUT/TEXTAREA/contenteditable. `Ctrl+K` must still fire in
  that case (so it works even when another field is focused). Resolution: check
  for the `focus-search` action (Ctrl+K) **before** the typing-field guard;
  all other shortcuts keep the guard.
- **Reorder scope:** persisted on drop (not visual-only) — confirmed. Visual
  during drag, `PUT` on drop.
- **Shuffle button placement:** only in `CollectionHeader` (Likes, Mix,
  Playlist). Charts/Artist are out of scope.

## Out of scope (future)

Crossfade/gapless, playback speed, sidebar playlist context menu, mini-mode,
sleep timer, recents view, help overlay, NowPlaying queue reorder, search
history dropdown.

## Testing

- **`reorderPlaylist` normalization/wiring:** the API method is a thin wrapper;
  unit-test the pure ordering helper if one is extracted, otherwise verify via
  tsc/lint + manual (DOM/IPC not node-testable). No new pure logic beyond the
  reorder-index math.
- **`shortcutAction`:** extend `src/ui/shortcuts.test.ts` (node, pure) to assert
  `Ctrl+K` → `{ type: 'focus-search' }` and that the mapping is unchanged
  otherwise.
- **Reorder index math:** if a pure `moveItem(array, from, to)` helper is
  extracted for the optimistic reorder, unit-test it (node, pure).
- **DOM-heavy wiring** (drag handlers, TopBar debounce, header button) verified
  by tsc + lint + manual smoke, consistent with the repo's node-only test env.

## Risks / notes

- Live-search debounce with `replace` avoids history spam; ensure the debounce
  is cancelled on unmount and on Enter (immediate navigate) to avoid a stale
  late navigation.
- Reorder persists via `PUT` of the full track-id list (same as add/remove);
  large playlists send the whole list — acceptable, matches existing behavior.
- Drag must not hijack the row's existing double-click-to-play / context menu;
  only the drag affordance initiates reordering.
